# Manual SQL migrations

These are SQL files that cannot be generated by `drizzle-kit` from the TypeScript schema. They live here, versioned in git, applied via `psql`.

## What goes here

- Extension enables (`CREATE EXTENSION`)
- Custom GUC parameter setup
- Triggers (UPDATE/DELETE prevention, etc.)
- RLS policies
- Raw SQL that's easier to maintain as SQL than wrap in Drizzle's `sql\`\`\`` helper
- One-off data migrations that need manual review

## What does NOT go here

Anything Drizzle can generate: tables, columns, indexes, foreign keys, simple defaults. Those live in the TypeScript schema files under `src/schema/` and are applied via `drizzle-kit push` (dev) or `drizzle-kit generate` + `drizzle-kit migrate` (staging/prod).

## Application order

Per Month 0 bootstrap:

1. Run `0001_enable_extensions.sql` FIRST — the schema depends on `vector` extension and `current_tenant_id()` function existing.
2. Run Drizzle-generated `0000_init.sql` SECOND (produced by `bun run db:generate` from `src/schema/`).
3. Run `0002_rls_policies.sql` THIRD — tables must exist before RLS policies reference them.
4. Run `0003_audit_triggers.sql` FOURTH — audit_log table must exist before triggers can attach.

For a fresh Neon dev branch:

```bash
# 1. Set admin connection
export DATABASE_URL_ADMIN="postgresql://bluecairn_admin:...@.../bluecairn"

# 2. Apply extension enables
psql "$DATABASE_URL_ADMIN" -f packages/db/migrations-manual/0001_enable_extensions.sql

# 3. Generate + apply Drizzle migration
cd packages/db && bun run db:generate && bun run db:push

# 4. Apply RLS policies
psql "$DATABASE_URL_ADMIN" -f packages/db/migrations-manual/0002_rls_policies.sql

# 5. Apply audit triggers
psql "$DATABASE_URL_ADMIN" -f packages/db/migrations-manual/0003_audit_triggers.sql
```

We'll wrap this in a single `bun run db:bootstrap` script once we've validated the sequence on a real Neon branch.

## Future migrations

When you add a new tenant-scoped table to `src/schema/`, remember to add its RLS policy as a new file:

```
migrations-manual/
  0001_enable_extensions.sql
  0002_rls_policies.sql          # initial policies
  0003_audit_triggers.sql
  0004_rls_for_vendors.sql       # new tenant-scoped table added → new policy
  0005_rls_for_inventory.sql
  ...
```

Each file is forward-only. If you made a mistake in `0002`, write `0004_fix_rls_X.sql` that corrects it — do not edit `0002`.

Per ENGINEERING.md § Migration hygiene: migrations are immutable once committed.
